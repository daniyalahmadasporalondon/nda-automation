How the system is wired, with real file:line and symbol references against the deployed code. Pick a topic on the left.

Architecture & request flow

The backend is a single Python process built on the standard-library HTTP server — no Django, no Flask, no ASGI. The frontend is plain static files served by the same process. There is no build step on either side.

The hand-rolled router

The HTTP layer is http.server from the standard library. The request handler is NdaAutomationHandler(SimpleHTTPRequestHandler) in nda_automation/server.py:332, run under stdlib ThreadingHTTPServer (one thread per request). The entrypoint is python -m nda_automation.server --host 0.0.0.0 --port <PORT>.

Dispatch is via four module-level exact-match route dictionaries that map a path string straight to a handler function — there is no decorator/registry framework:

Dictserver.pyHolds
_GET_EXACT_ROUTES~224Authenticated read endpoints.
_PUBLIC_GET_EXACT_ROUTES~269Public GET (/login, /api/auth/status, OAuth callbacks).
_POST_EXACT_ROUTES~276Authenticated POST (/api/review, /api/matters, /api/generate-nda, …).
_PUBLIC_POST_EXACT_ROUTES~319Public POST (/api/auth/logout only).

The dispatchers:

  • _handle_get() (server.py:343, used by do_GET + do_HEAD) — host check → static-file serving (before auth) → public routes → _authorize_request() → rate-limit → exact GET routes → pattern routes for /api/matters/<id>/… → 404.
  • do_POST() (server.py:449) — host check → CSRF check before auth → public POST → auth → rate-limit → exact POST routes → matter-pattern POST routes.
  • do_DELETE() (server.py:524) — host → CSRF → auth → matter-pattern DELETE routes.

Per-matter sub-resources (/api/matters/<id>/review-refresh, …/working-docx, etc.) are not in the dicts; they are matched by suffix in the fallback blocks of each dispatcher.

The routes/ package

Handler functions live in nda_automation/routes/ — one module per surface, imported at the top of server.py (~lines 81-103) and referenced directly in the route dicts. The modules are: admin.py, approval.py, auth.py, common.py, corpus.py, dashboard.py, docusign.py, drive.py, entities.py, generation.py, gmail.py, matters.py, pdf_markup.py, playbook.py, review.py, send_document.py. A handler takes the request handler (self) and writes its own response; there is no middleware stack.

Static files

The /static/* branch in _handle_get() (server.py:356-364) resolves the requested path, asserts it stays inside STATIC_DIR = ROOT/"static" (server.py:110) as a path-traversal guard, and streams the file. This is why both guides are served with no per-file route: dropping static/developer-guide.html into the tree is enough to serve it at /static/developer-guide.html.

Cache-bust tokens & the asset manifest

Every script/stylesheet is loaded from index.html with a ?v=YYYYMMDD<letter> token so browsers reload it after a change. With immutable caching live, a forgotten token bump would pin a stale asset in a user's browser for up to a year — so a committed manifest guards against it:

  • The manifest — static/asset-tokens.json records {path: {v, sha256}} for every versioned asset. It's the CI staleness guard's source of truth.
  • The tool — python -m nda_automation.static_versioning (nda_automation/static_versioning.py). --check fails if any asset's bytes changed without its ?v= token (and manifest entry) changing; --write regenerates the manifest after you've bumped the tokens.
  • The test — tests/test_static_asset_manifest.py runs the same check in CI, so a bytes-changed-but-token-unchanged slip is a red build, not a silent stale-cache incident.
The edit ritual for any static/ change: edit the file → bump its ?v= token in index.html → run python -m nda_automation.static_versioning --write → confirm --check passes. scripts/cache_bust_guard.py --tree is the belt-and-braces tree-wide guard.

Syntax floor (the shipped bundle must parse on old webviews)

Everything under static/**/*.{js,mjs} ships untranspiled and most loads as classic <script> tags in one chain — a single SyntaxError detonates the whole chain (app.js and all it wires) at parse time on the oldest webview a user opens, before any feature detection can run. tests/frontend/syntax-floor.cjs enforces the floor (Safari 13.x era): full ES2020 syntax and numeric separators are allowed; logical assignment (??=/||=/&&=) and anything ES2022+ (class fields, #private, static blocks, top-level await) are rejected — the parse runs at ecmaVersion 2021 with a dedicated AST scan for the logical-assignment operators acorn would otherwise accept.

The data dir & deploy shape

All durable state roots at NDA_DATA_DIR (read in matter_store.py:24; default ROOT/"data"). In production it is the Render persistent disk at /var/data, which holds matters/*.json, uploads/, the lock file, Gmail cursors, and the live playbook.json + sidecars. The app ships as a Docker image (python:3.11-slim + LibreOffice) on Render Standard. See Deploy & Ops for the full env-var map.

Mental model. A request walks: host check → (CSRF for writes) → auth → rate-limit → an exact-match function → a JSON or file response. Everything stateful is a JSON file under NDA_DATA_DIR. Everything user-facing is a static file under static/.

The Playbook as the source of truth

The north star of this codebase: playbook.json is the single source for all clause rules, approved options, and thresholds. The engine and the AI infer, judge, and apply from it — they never hardcode a rule that the playbook also owns. Duplicated/hardcoded rules are treated as "deterministic ghosts" to eliminate.

Where it lives & how it loads

  • Bundled copy: playbook.json in the repo root.
  • Live copy: $NDA_DATA_DIR/playbook.json (e.g. /var/data/playbook.json). On first run, if the persistent copy is absent the bundled one is seeded once; it is never clobbered on redeploy.
  • Loader: load_playbook() in nda_automation/checker.py:137, resolving PLAYBOOK_PATH at startup (persistent disk if set, else bundled).
  • Sidecars in the same dir: playbook.draft.json, playbook.runtime.json, playbook.history.json.
Test gotcha. Patching PLAYBOOK_PATH does not redirect the playbook helpers — inject an explicit path/runtime in tests instead.

Clause shape

A clause object in the "clauses" array carries id, name, a "type" of "required" or "prohibited", a requirement, and a rules block (version, clause_type, acceptable_position, pass_conditions, fail_conditions, review_triggers, evidence_requirements, redline_guidance), plus search_terms, taxonomy_groups, role_terms, and one_way_terms where relevant. Rules are indexed by the checker on load — no build step. (Native vs dynamic clauses are an engine distinction, not a JSON key — see the recipe for adding one.)

The draft / publish API

Authoring is admin-gated and routed through nda_automation/routes/playbook.py:

EndpointHandler (routes/playbook.py)Purpose
POST /api/playbook/drafthandle_playbook_validate_draft (~74)Returns {valid, errors, warnings}; warnings are advisory.
POST /api/playbook/draft/savehandle_playbook_draft_save (~89)Persists the draft sidecar.
POST /api/playbook/publishhandle_playbook_publish (~115)Lint-gated promotion of draft → live.

The lint gate is _enforce_playbook_lint() in nda_automation/playbook_authoring.py, which calls lint_violations_for(playbook, severities={"error"}) — only error-severity violations block publish; warnings surface in draft validation but never wedge a publish. The deterministic lint covers decision-space, condition-well-formedness, redline-template, approved-options, and referential integrity. An optional AI semantic lint (advisory, default-off) layers on top.

Error contract: failures raise PlaybookAuthoringError(payload, status=400) (playbook_authoring.py), surfacing as 400 {"error": "…"} with an optional "code" (e.g. playbook_draft_conflict). The lint is fail-open: if the lint itself throws, the publish is not blocked.

The contract. To change how NDAs are judged, you edit the Playbook — never the engine. The AI receives the playbook rules in its packet and is told to comply to them; it must never invent or override a rule.

The AI review pipeline

Review is on-demand only — there is no automatic AI review on inbound documents. A user clicks Review; the server enqueues an async job; the workstation polls for the result.

Entry point & on-demand contract

The review is triggered by POST /api/matters/<id>/review-refresh → handle_matter_review_refresh() in nda_automation/routes/matters.py:119. It enqueues a review and returns 202 immediately — it never runs the AI engine inline (asserted by tests/test_async_review_backend.py). Inbound Gmail matters import as "Not Reviewed" and are reviewed only when a human asks (server.py:1134 comment; tests/test_inbound_auto_review.py).

The packet

Before the model is called, the app assembles an assessment packet via build_ai_assessment_packet() in nda_automation/ai_assessment_prompt.py:254. It bundles: the clause requirements / playbook rules (playbook_rules_for_ai() in playbook_rules.py), the binding policy block (build_playbook_policy_block() in playbook_policy.py), the Stage-1 structure context (build_contract_structure() in contract_structure.py), and the source paragraphs (with omitted-paragraph counts + truncation markers). The model sees a flat {id, index, text} packet plus the rules.

The two models

Reviewer (first pass)
Default anthropic/claude-opus-4.8-fast — DEFAULT_OPENROUTER_MODEL at ai_review.py:28, selected by env NDA_AI_MODEL (AI_REVIEW_ENV_MODEL, ai_review.py:35). Resolved in configured_ai_assessment_reviewer() (ai_assessor.py:422).
Verifier (second pass)
Default deepseek/deepseek-v4-pro — DEFAULT_VERIFIER_MODEL at ai_verifier.py:61, env NDA_AI_VERIFIER_MODEL (ai_verifier.py:63). Enabled by NDA_AI_VERIFIER (opt-in). Resolved in resolve_verifier() (ai_verifier.py:1148).

Both run through OpenRouter; the key (OPENROUTER_API_KEY) lives in backend config and is never exposed to the browser.

The decision arbiter

Clause decisions are synthesised in three phases:

  1. AI assessment → build_ai_first_review_result() (ai_first_review.py:327) normalises reviewer assessments into clause results.
  2. Adversarial verifier → apply_ai_verifier() (ai_verifier.py:128). Escalated clauses (fail/review) are always verified; passes are verified only when confidence is below the high-confidence pass threshold (0.85, ai_verifier.py:92). A refute/uncertain verdict downgrades to review — the verifier never autonomously acquits a clause to pass.
  3. Reconciliation → _reconcile_unchanged_clause_review_states() (ai_first_review.py:809) and _reconcile_clause_decision() (ai_first_review.py:857) propagate changes into review_state and apply floor normalisations.

The semantic confidence floor is SEMANTIC_REVIEW_THRESHOLD = 0.75 in decision_arbiter.py:32 — a pass below it is escalated to review. review_state.py:332 (_normalize_clause_decision()) imports the same constant so the floor is single-sourced.

Evidence grounding

Every pass/fail on a present clause must cite a real, verbatim quote. classify_grounding() (evidence_grounding.py:88) returns grounded / absence / ungrounded. Ungrounded findings are demoted by downgrade_ungrounded_finding() (evidence_grounding.py:223) → decision REVIEW, blocks_send=True, reason code ungrounded_finding. The two legitimate no-quote cases: a required clause FAILed as missing, and a prohibited clause that correctly does not appear. Wired in via build_grounding() (ai_first_review.py:699).

review_state

The per-clause review_state struct (review_state.py:40) carries state (pass|review|check), decision (pass|review|fail), blocks_send, blocks_auto_send, requires_human_review, and reason_codes. clause_review_state() sets it per clause; aggregate_review_state() (review_state.py:113) rolls up the document-level state.

Why it's trustworthy. Fail-closed everywhere: a missing assessment, an invalid citation, or a provider error escalates to Review and blocks auto-send rather than waving a document through. Low confidence becomes "ask a human"; uncertainty leans to Review.

Redlines & generation

Two write paths produce documents: redlining marks up a received NDA as Word tracked changes; generation builds a brand-new NDA deterministically from the template + playbook.

Redlines: propose → diff → anchor → export

  • Propose. A failed clause's suggested-fix wording comes from the playbook's redline_guidance; the FE edit model lives in static/js/modules/redline-edit-contract.mjs (action types REPLACE_PARAGRAPH / DELETE_PARAGRAPH / INSERT_AFTER_PARAGRAPH / FORMAT_PARAGRAPH).
  • Diff. Word tracked changes are built in nda_automation/redline_xml.py — the _source_tracked_replace_paragraph* / _source_tracked_delete_paragraph / _source_tracked_insert_paragraphs builders emit <w:ins> / <w:del> revision markup. redline_replace_uses_whole_text_markup() (redline_xml.py:814) decides whole-text vs char-level tokenised diff.
  • Anchor. source_index is non-unique provenance — split blocks (a paragraph cut on an internal blank line) share it on purpose. paragraph_index (a.k.a. index) is the unique blank-line-block key. Export and the health gate key on paragraph_index, falling back to source_index: see _expected_redline_source_index() (docx_health.py:793) and redline_source_index() (redline_edit_contract.py).
  • Export. Built in docx_export.py.

Coverage gates

The DOCX gate verify_export_content_coverage() (docx_health.py:~150) asserts the export body is non-empty, covers ≥80% of source text, and that the accepted-paragraph sequence matches the expected sequence built from source + redline edits. The PDF path can't do exact sequence comparison (reconstruction reflows), so verify_pdf_reconstruction_redline_coverage() (docx_health.py:~203) instead finds each redline's target paragraph by confident text match and verifies w:ins/w:del markup is present, using one-to-one bipartite matching so identical redlines consume distinct paragraphs. Both fail closed.

Generation (deterministic)

POST /api/generate-nda (routed at server.py:282) delegates to the generation workflow. By default generation is deterministic: generation_ai_enabled() (nda_generation.py) reads NDA_GENERATION_AI_ENABLED; in production it is set to false, so no OpenRouter call is made and the document is built purely from the template + playbook. The template is nda_automation/templates/generic_nda.docx (Aspora = second party, counterparty = first), with Term/Survival and Confidential-Information exclusions realigned from the template defaults to the live playbook at generation time.

gen-verify (the ship gate)

Before any persistence, assert_generated_nda_is_on_position() (nda_generation.py:~980) runs two independent screens: (1) self_check_generated_nda() (nda_generation.py:1094) — a network-free deterministic oracle that re-extracts the generated text and runs review_nda(..., ai_enabled=False), requiring zero native-clause failures, plus a key-free stub check that non_circumvention does not fail; and (2) a prohibited-position scan over the whole document. Either failure raises NdaGenerationError and blocks the ship.

The governing-law lock

Governing law is fixed by the signing entity and cannot be overridden. In nda_generation.py:~893, an override_id that differs from the entity default raises NdaGenerationError (input category); an override equal to the default is a harmless no-op. The manifest always records governing_law_overridden = False. This closes the divergence where an override could pull another entity's court into a signed NDA.

PDF / DOCX rendering & Approach C

"Approach C" converts a PDF-source NDA into a working DOCX at ingest time, so the whole redline/export pipeline operates on a real Word document instead of a reconstructed one on the fly. A vendored faithful-render path shows the original DOCX byte-for-byte in the browser.

Approach C: convert at ingest

  • convert_pdf_matter_to_docx() lives in nda_automation/pdf_ingest_conversion.py:65.
  • It is wired at ingest: ingestion_service.py:1321 → _convert_pdf_matter_at_ingest() → _persist_pdf_working_conversion() (ingestion_service.py:1367) → convert_pdf_matter_to_docx(). Non-PDF matters are skipped (document_type guard). The conversion re-keys paragraphs to anchor by source_index and registers a role="working" artifact.
  • Fail-open: any error in the chain returns the original matter unchanged — conversion never hard-blocks ingest.

working_docx_ready & the /working-docx seam

  • working_docx_ready is set in the render payload at matter_render_job.py:91 via matter_has_working_docx() (matter_render_job.py:95), which checks for a ROLE_WORKING artifact in the registry.
  • The route GET /api/matters/<id>/working-docx → handle_matter_working_docx() (routes/matters.py:588, wired at server.py:390) streams the working DOCX bytes via pdf_export_service.build_matter_working_docx_export().

The vendored faithful render (docx-preview)

For DOCX-source matters there is a byte-faithful browser render, vendored (no CDN):

  • static/vendor/docx-preview/docx-preview.min.js (+ LICENSE) — confirmed present in the deployed tree.
  • static/js/docx-faithful-render.js — lazily injects jszip + docx-preview on first render (retrying), exposes renderFaithfulDocx(container, options), and always returns { ok, reason } — never throws, degrading to reconstruction on any failure.
  • Flag: nda.faithfulDocxRender (localStorage key from static/js/config.js:20; gate faithfulDocxRenderEnabled() in docx-faithful-render.js:61). Default OFF — set localStorage["nda.faithfulDocxRender"] = "1" to enable.

reconstruct_pdf_to_docx (the low-level fallback)

reconstruct_pdf_to_docx() (pdf_docx_reconstruction.py:318) is the engine that turns PDF bytes into DOCX bytes (semaphore/slot backpressure, page-cap, owner-keyed LRU cache, subprocess timeout/RLIMIT hardening). It knows nothing about redlines. convert_pdf_matter_to_docx() is the high-level wrapper that calls it, re-keys paragraphs for anchor-by-index, and enforces an empty-body guard.

PDF extraction: pypdf vs PyMuPDF

Document structure/text extraction uses pypdf (extract_pdf_document() in pdf_text.py:140). PyMuPDF / fitz is used only for visual signals — a cheap image-count decompression-bomb guard (pdf_text.py:280, via get_image_info(), no pixel decode) and page rasterization for preview thumbnails. fitz is never used for structure.

Why the Dockerfile asserts fitz. A silent PyMuPDF ABI mismatch would break page-image rendering only at runtime. The build smoke-test (import fitz; fitz.open(), Dockerfile:43) fails the build loudly instead. See Deploy & Ops.

Data model & security

The unit of work is a matter — one NDA and everything attached to it, persisted as a JSON record under NDA_DATA_DIR. Multi-tenant isolation is fail-closed.

The matter store

  • Module: nda_automation/matter_store.py. create_matter() (matter_store.py:1020) mints matter_<uuid12> and writes $NDA_DATA_DIR/matters/<id>.json.
  • Locking: _locked_store() (matter_store.py:1124) takes an in-process threading.RLock() (30s timeout) and a cross-process fcntl.flock() on matters.lock, so create/dedupe/update read-modify-write is serialised even across processes. The one historical bug was an unlocked dedupe pre-check; the real R-M-W path was always sound.
  • Dedup: _find_gmail_duplicate_unlocked() (matter_store.py:1615) runs inside the lock — it indexes attachment id / case-folded filename / SHA256, gates each candidate through ownership, and confirms filename-only matches by hash. Atomic check+write closes the TOCTOU window.

Performance & observability (scale hardening)

The store is a single-process file store, so a growing corpus turned several hot paths O(store). Four changes keep them bounded:

  • Per-record list cache — list_matters() is cached under a fingerprint of every record file's (name, mtime, size), consulted/refreshed only while the store lock is held (so it observes a consistent set) and storing the UNFILTERED list — owner scoping is always applied per-call after, so the cache can never serve cross-tenant data. The blob is keyed per record filename, not one monolithic blob: a write-through patch updates the one written record's entry (O(one record)) instead of re-serialising the whole store (~330ms/write on a 47MB/1200-matter store → a few ms). NDA_DISABLE_MATTER_LIST_CACHE forces the uncached path for tests.
  • Timeline cap — matter_timeline was append-only and grew unbounded; capped_timeline() now clamps it to TIMELINE_MAX_EVENTS = 500 (newest kept, oldest dropped, with a marker event) on every write.
  • Corpus fingerprinting off the request path — dedup SimHash fingerprints are computed inline only under a FINGERPRINT_INLINE_COMPUTE_BUDGET per request; the remainder is computed by a single-flight background backfill (_start_fingerprint_backfill, corpus_index.py), so a large cold store never serialises O(store) SimHash computes onto one request.
  • Failure visibility — swallowed failures now log (module LOGGER.warning/error with the exception type) instead of failing silently — e.g. the PyMuPDF import path logs the exact module+message at ERROR on a genuine load failure rather than degrading rendering invisibly.

Artifacts

nda_automation/artifact_registry.py is a thin registry over a matter's documents. Roles (artifact_registry.py:39-47): original, redline, reviewed, generated, counter, sent, signed, and working (the Approach-C PDF→DOCX). Actors: counterparty, ai, human (or an entity id). The matter carries an artifacts list + a current_artifact_id pointer.

Fail-closed ownership

_matter_owner_matches() (matter_store.py:1666) is the tenant guard, used by every matter op (list/get/update/delete/export/dedup):

  • Empty caller owner_user_id = single-tenant/no-auth → wildcard (all matters in scope).
  • Non-empty caller = authenticated → matter's owner_user_id must match exactly.
  • A matter with no owner field (legacy / pre-ownership Gmail import) is not a wildcard — it fails closed and is never served to an authenticated caller with a different owner.

Auth, admin & CSRF

  • Auth: _authorize_request() (server.py:740) — Google OAuth session first, then HTTP Basic (NDA_AUTH_USERNAME/NDA_AUTH_PASSWORD), else 401. Loopback hosts are auth-exempt for local dev.
  • Admin: request_is_admin() (http_auth.py:20). Authority = the NDA_ADMIN_USERS env list (matched against the verbatim user id, e.g. google:<sub>, and — for Google callers only — the OAuth-verified email), plus an in-app persisted admin list (Google callers only). Basic-auth usernames never inherit admin from an email.
  • CSRF: origin_allowed_for_request() (csrf.py:20) runs for state-changing methods only, matching the Origin (or Referer) host against an allow-set. Enforcement flag NDA_ENFORCE_CSRF (default ON for non-loopback binds, OFF for localhost). Checked before auth in do_POST/do_DELETE.
  • Rate limit: rate_limit.py + _rate_limit_request() (server.py:809) — bucketed by user id then real client IP (proxy-aware via NDA_TRUSTED_PROXY_COUNT), 300/min general, 240/min render, 60s window, 429 + Retry-After.

Gmail intake · DocuSign · Outbound

Three external surfaces: the Gmail poller that auto-imports inbound NDAs, the DocuSign e-signature flow, and the outbound send paths (with a hard recipient-confirmation gate).

Gmail intake

  • Scheduler: _gmail_sync_scheduler_loop() (server.py:957), a daemon thread; each tick runs _run_scheduled_gmail_sync() (server.py:1113). Master kill-switch env NDA_GMAIL_SYNC_ENABLED (server.py:992).
  • Fetch query (fixed structure) + sender exclusion: _inbound_envelope_query(90) (gmail_integration.py:185) → in:inbox has:attachment (filename:docx OR filename:pdf) -from:me newer_than:90d, with a -from: exclusion clause for noise senders appended to the query. The default excluded set (DEFAULT_GMAIL_INBOUND_EXCLUDED_SENDERS, app_settings.py:158) is the e-sign platforms (docusign.net/.com, adobesign.com, documents.adobe.com, echosign.com, hellosign.com, pandadoc.com) plus calendar-notification@google.com; it's admin-editable, but DOCUSIGN_NOTIFICATION_DOMAINS is a hard floor that always applies. Window default GMAIL_INBOUND_WINDOW_DAYS = 90 (admin-configurable). Per-poll limit default 20, hard cap 40 (_MAX_GMAIL_IMPORT_LIMIT_CLAMP, gmail_integration.py:229; env NDA_GMAIL_IMPORT_LIMIT).
  • Deterministic pre-gate (before the paid model): a content scorer drops obvious non-NDAs (proposals, invoices, decks, questionnaires) before a paid intake AI call, cutting intake-AI spend markedly. A mandatory escape hatch (_metadata_has_explicit_nda_signal / the filename+text NDA signal, gmail_integration.py:1582) forces any document with an explicit NDA signal through to the AI regardless — including foreign-language NDA titles. Detection uses a fixed built-in vocabulary; an opt-in custom-terms extension is gated behind NDA_GMAIL_CUSTOM_TERMS_ENABLED (default off; _custom_detection_terms only adds content-scorer signals — never widens the fetch query).
  • Executed-NDA capture: e-sign platform notification mail is usually terminal-dropped, but a completion email is often the only copy of an executed NDA. When such a platform email carries an explicit NDA signal (_esign_notification_nda_hit, gmail_integration.py:1220), it's let through the pipeline but clamped to at most the triage lane and stamped with the esign_notification_nda provenance (ESIGN_NDA_CAPTURE_TRIAGE_REASON) — never a silent auto-import. Gated by NDA_GMAIL_ESIGN_NDA_CAPTURE (default on; off restores the unconditional drop).
  • Intake classifier (NDA vs not): classify_intake_attachment() (gmail_intake_classifier.py:149); model env NDA_GMAIL_INTAKE_MODEL, default deepseek/deepseek-v4-flash (gmail_intake_classifier.py:45). Additive — any error/timeout/cap falls back to the deterministic lane classifier.
  • Attachment selector / triage: gmail_attachment_selector.py; model env NDA_GMAIL_TRIAGE_MODEL, default deepseek/deepseek-v4-pro (line 16). Injection defense: the model's selection is intersected with the real candidate attachment ids (gmail_attachment_selector.py:101-109) — an injected instruction can't import an attachment that was never a candidate.
  • Review is deferred. import_inbound_matters() (gmail_integration.py:541) fetches, triages, and classifies but does not enqueue AI review — inbound matters land "Not Reviewed" and are reviewed only on demand.

DocuSign

nda_automation/docusign_integration.py holds the client. HttpDocuSignClient (line 404) creates envelopes (create_envelope(), ~431), reads status (get_envelope_status()) and recipients (get_envelope_recipients()). Signature placement uses anchor tabs built in _tabs_for(signer) (~305) — signHereTabs + dateSignedTabs anchored to document strings. Two signers: the counterparty and the single Aspora signer (NDA_DOCUSIGN_ASPORA_SIGNER_NAME/_EMAIL). OAuth env (docusign_connection.py:48-54): NDA_DOCUSIGN_CLIENT_ID, _CLIENT_SECRET, _OAUTH_REDIRECT_URI, _AUTH_SERVER (demo|prod), _CONNECT_HMAC_KEY.

Envelope document = current reviewed DOCX (P0 invariant). docusign_workflow.py (~531) never signs the un-redlined original when a matter has reviewer edits. It rebuilds the reviewed DOCX from the matter's CURRENT state via the same build_reviewed_docx path (and the same coverage gates) the download + approval handlers use — deliberately not trusting a stored role=reviewed artifact, which could be stale after a reversal (_DOCUMENT_PRECEDENCE excludes ROLE_REVIEWED as a backstop). If a matter has edits but the reviewed DOCX can't be produced, the send is refused ("Refusing to send the un-redlined original for signature.") rather than sending the original. Edit-free matters (comment-only/clean) fall through to precedence-ordered document selection.

Outbound (send redline / send document)

  • Send redline: handle_gmail_send_redline() (routes/gmail.py:397, wired at server.py:287).
  • Send document: handle_send_document() (routes/send_document.py:33, wired at server.py:285) — reuses the same Gmail send plumbing.
  • Recipient-confirmation gate: require_confirmed_recipient() (gmail_matter_outbox.py:103-118), reached via validate_outbound_send_ready() before any Gmail call. It raises RecipientConfirmationError → 400 if the recipient came from an inbound header and was not confirmed, if the confirmed value is empty, or if it doesn't match the resolved recipient ("The confirmed recipient does not match the NDA recipient; refusing to send."). This blocks spoofed-Reply-To exfiltration.

Admin / operator endpoints (no UI)

  • Bulk-archive Gmail noise — POST /api/admin/matters/bulk-archive (routes/admin.py:1224 → matter_store.bulk_archive_gmail_matters). Admin-gated. Body takes an explicit owner_user_id (never inferred), a required created_after/created_before window, dry_run (default true), and a confirm sha256 of the current selection. Execute requires dry_run:false + a matching confirm hash (stale hash → 409 with the fresh hash), and refuses unless inbound Gmail import is disabled (a ledger-flush race with a live import → 409). Only predicate ∩ confirmed is deleted; every record + source doc is archived to pruned-matters/ before deletion; deleted message ids are marked in the per-owner processed ledger so the next poll won't re-import them.
  • Backup export — GET /api/matters/export (routes/admin.py:862, handle_matter_backup). Admin-gated; dumps full extracted NDA text + metadata as JSON. Defaults to the caller's owner scope; ?owner=__all__ exports all owners (export_all_matters_backup()), stamping scope: "all-owners".
Disk-durability blind spot. The boot guard checks NDA_DATA_DIR is set, not that the disk is actually mounted — an unmounted /var/data can pass the guard. Confirm the disk is attached before trusting persistence.

How-to recipes

Common changes, with the exact place to make them.

Add a clause type

  1. Add the clause object to the "clauses" array in playbook.json (id, name, type required|prohibited, requirement, rules block, search_terms, taxonomy_groups). The checker indexes it on load.
  2. Native vs dynamic: native clauses have a deterministic engine checker; dynamic clauses (e.g. non_circumvention) are only emitted by the AI engine — the deterministic engine never produces them, so a dynamic clause needs both the playbook entry and an AI-engine seam (and any test must add a backend seam, not just a FE change).
  3. Validate via POST /api/playbook/draft, then publish via POST /api/playbook/publish (lint-gated, error-severity blocks).

Change an AI model

Set the env var and redeploy (or flip live in Render):

JobEnv varDefault

Change a playbook rule

Edit the rule in the Playbook, never the engine. The AI receives the playbook rules in its packet; deterministic checkers read the same playbook. Hardcoding a rule the playbook owns is a "deterministic ghost" — don't.

Debug a stuck review

  1. Remember review is async — POST /api/matters/<id>/review-refresh returns 202; the result arrives later. "Reviewing…" is normal up to ~4 min; a stale recorded start (~10 min) reads as "stalled", not failed.
  2. Only a recorded error shows "failed". Check the provider key (OPENROUTER_API_KEY), NDA_AI_REVIEW_ENABLED, and the review-state reason codes.
  3. Grounding demotions (ungrounded_finding) and the 0.75 confidence floor push borderline clauses to Review by design — that's not a bug.

Backend: python3 -m pytest -q
Frontend: npm run test:frontend:utils && npm run test:frontend
Lint: ruff check nda_automation tests

There is no Makefile; pytest + npm run directly. The pytest suite includes the review eval gate (counsel-style review cases).
