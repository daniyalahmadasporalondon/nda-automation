Quick start — run it locally

Zero build step, one Python process. From a fresh clone you are signed in and reviewing NDAs in about five commands. There is no Node build and no database — durable state is JSON files under a data dir.

The five commands

Requires Python 3.9+. DOCX review/export work out of the box (python-docx is a core dep); PDF and Gmail are optional extras.

git clone <repo-url> && cd nda-automation
python3 -m pip install -e ".[pdf,gmail]"
cp .env.example .env  (then fill in keys — .env is gitignored)
set -a; source .env; set +a
python -m nda_automation.server --port 8787

Then open http://127.0.0.1:8787.

The extras matrix

The optional-dependency extras are declared in pyproject.toml ([project.optional-dependencies]). Pick the set you need:

ExtraAddsWhen
.[pdf]pypdf + PyMuPDF + pdf2docxPDF intake & PDF→DOCX reconstruction.
.[gmail]Google API clientThe Gmail inbound connector.
.[pdf,gmail]Both of the aboveWhat the Render blueprint installs — the usual local set.
.[pdf,gmail,tables]+ camelot (2.x) table extraThe full prod-Docker set (adds NDA_TABLE_AUGMENTATION_ENABLED backends). Heavier resolve — the PyMuPDF pin exists so this co-resolves.
Without .[pdf], PDF uploads fail with a clear "PDF support is not installed" error, and PDF→DOCX reconstruction is unavailable — download/export routes say so explicitly rather than serving extracted text as a fake Word conversion. The prod Docker image installs the full .[pdf,gmail,tables] set; the docker-build CI job is what catches a co-resolution/ABI break there before it ships.

Minimum .env to enable AI review

The cp .env.example .env step seeds a full annotated file; the review-relevant floor is:

NDA_AI_REVIEW_ENABLED=true
NDA_AI_PROVIDER=openrouter
NDA_AI_MODEL=x-ai/grok-4.3  (the shipped local default; see the divergence note below)
OPENROUTER_API_KEY="your-openrouter-api-key"

Local reviewer model ≠ prod. .env.example deliberately ships the cheaper x-ai/grok-4.3 for local dev, while the code default and prod both run anthropic/claude-opus-4.8-fast. If a local review disagrees with prod on a borderline clause, that is the model difference, not a bug — set NDA_AI_MODEL to the Opus model to mirror prod. (Fuller treatment in AI Review Pipeline and the Change an AI model recipe.)

Architecture & request flow

The backend is a single Python process built on the standard-library HTTP server — no Django, no Flask, no ASGI. The frontend is plain static files served by the same process. There is no build step on either side.

The hand-rolled router

The HTTP layer is http.server from the standard library. The request handler is NdaAutomationHandler(SimpleHTTPRequestHandler) in nda_automation/server.py, run under stdlib ThreadingHTTPServer (one thread per request). The entrypoint is python -m nda_automation.server --host 0.0.0.0 --port <PORT>.

Dispatch is via four module-level exact-match route dictionaries that map a path string straight to a handler function — there is no decorator/registry framework:

Dictserver.pyHolds
_GET_EXACT_ROUTES~224Authenticated read endpoints.
_PUBLIC_GET_EXACT_ROUTES~269Public GET (/login, /api/auth/status, OAuth callbacks).
_POST_EXACT_ROUTES~276Authenticated POST (/api/review, /api/matters, /api/generate-nda, …).
_PUBLIC_POST_EXACT_ROUTES~319Public POST (/api/auth/logout only).

The dispatchers:

  • _handle_get() (server.py, used by do_GET + do_HEAD) — host check → static-file serving (before auth) → public routes → _authorize_request() → rate-limit → exact GET routes → pattern routes for /api/matters/<id>/… → 404.
  • do_POST() (server.py) — host check → CSRF check before auth → public POST → auth → rate-limit → exact POST routes → matter-pattern POST routes.
  • do_DELETE() (server.py) — host → CSRF → auth → matter-pattern DELETE routes.

Per-matter sub-resources (/api/matters/<id>/review-refresh, …/working-docx, etc.) are not in the dicts; they are matched by suffix in the fallback blocks of each dispatcher.

The routes/ package

Handler functions live in nda_automation/routes/ — one module per surface, imported at the top of server.py (~lines 81-103) and referenced directly in the route dicts. The modules are: admin.py, approval.py, auth.py, common.py, corpus.py, dashboard.py, docusign.py, drive.py, entities.py, generation.py, gmail.py, matters.py, pdf_markup.py, playbook.py, review.py, send_document.py. A handler takes the request handler (self) and writes its own response; there is no middleware stack.

Static files

The /static/* branch in _handle_get() (server.py) resolves the requested path, asserts it stays inside STATIC_DIR = ROOT/"static" (server.py) as a path-traversal guard, and streams the file. This is why both guides are served with no per-file route: dropping static/developer-guide.html into the tree is enough to serve it at /static/developer-guide.html.

Cache-bust tokens & the asset manifest

Every script/stylesheet is loaded from index.html with a ?v=YYYYMMDD<letter> token so browsers reload it after a change. With immutable caching live, a forgotten token bump would pin a stale asset in a user's browser for up to a year — so a committed manifest guards against it:

  • The manifest — static/asset-tokens.json records {path: {v, sha256}} for every versioned asset. It's the CI staleness guard's source of truth.
  • The tool — python -m nda_automation.static_versioning (nda_automation/static_versioning.py). --check fails if any asset's bytes changed without its ?v= token (and manifest entry) changing; --write regenerates the manifest after you've bumped the tokens.
  • The test — tests/test_static_asset_manifest.py runs the same check in CI, so a bytes-changed-but-token-unchanged slip is a red build, not a silent stale-cache incident.
The edit ritual for any static/ change: edit the file → bump its ?v= token in index.html → run python -m nda_automation.static_versioning --write → confirm --check passes. scripts/cache_bust_guard.py --tree is the belt-and-braces tree-wide guard.

Syntax floor (the shipped bundle must parse on old webviews)

Everything under static/**/*.{js,mjs} ships untranspiled and most loads as classic <script> tags in one chain — a single SyntaxError detonates the whole chain (app.js and all it wires) at parse time on the oldest webview a user opens, before any feature detection can run. tests/frontend/syntax-floor.cjs enforces the floor (Safari 13.x era): full ES2020 syntax and numeric separators are allowed; logical assignment (??=/||=/&&=) and anything ES2022+ (class fields, #private, static blocks, top-level await) are rejected — the parse runs at ecmaVersion 2021 with a dedicated AST scan for the logical-assignment operators acorn would otherwise accept.

The data dir & deploy shape

All durable state roots at NDA_DATA_DIR (read in matter_store.py; default ROOT/"data"). In production it is the Render persistent disk at /var/data, which holds matters/*.json, uploads/, the lock file, Gmail cursors, and the live playbook.json + sidecars. The app ships as a Docker image (python:3.11-slim + LibreOffice) on Render Standard. See Deploy & Ops for the full env-var map.

Mental model. A request walks: host check → (CSRF for writes) → auth → rate-limit → an exact-match function → a JSON or file response. Everything stateful is a JSON file under NDA_DATA_DIR. Everything user-facing is a static file under static/.

The NDA lifecycle end to end

The whole system exists to walk one NDA through seven stages. The Playbook feeds the clause-review stage its rules; every downstream stage inherits that judgement.

The Playbook as the source of truth

The north star of this codebase: playbook.json is the single source for all clause rules, approved options, and thresholds. The engine and the AI infer, judge, and apply from it — they never hardcode a rule that the playbook also owns. Duplicated/hardcoded rules are treated as "deterministic ghosts" to eliminate.

Where it lives & how it loads

  • Bundled copy: playbook.json in the repo root.
  • Live copy: $NDA_DATA_DIR/playbook.json (e.g. /var/data/playbook.json). On first run, if the persistent copy is absent the bundled one is seeded once; it is never clobbered on redeploy.
  • Loader: load_playbook() in nda_automation/checker.py, resolving PLAYBOOK_PATH at startup (persistent disk if set, else bundled).
  • Sidecars in the same dir: playbook.draft.json, playbook.runtime.json, playbook.history.json.
Test gotcha. Patching PLAYBOOK_PATH does not redirect the playbook helpers — inject an explicit path/runtime in tests instead.

Clause shape

A clause object in the "clauses" array carries id, name, a "type" of "required" or "prohibited", a requirement, and a rules block (version, clause_type, acceptable_position, pass_conditions, fail_conditions, review_triggers, evidence_requirements, redline_guidance), plus search_terms, taxonomy_groups, role_terms, and one_way_terms where relevant. Rules are indexed by the checker on load — no build step. (Native vs dynamic clauses are an engine distinction, not a JSON key — see the recipe for adding one.)

The draft / publish API

Authoring is admin-gated and routed through nda_automation/routes/playbook.py:

EndpointHandler (routes/playbook.py)Purpose
POST /api/playbook/drafthandle_playbook_validate_draftReturns {valid, errors, warnings}; warnings are advisory.
POST /api/playbook/draft/savehandle_playbook_draft_savePersists the draft sidecar.
POST /api/playbook/publishhandle_playbook_publishLint-gated promotion of draft → live.

The lint gate is _enforce_playbook_lint() in nda_automation/playbook_authoring.py, which calls lint_violations_for(playbook, severities={"error"}) — only error-severity violations block publish; warnings surface in draft validation but never wedge a publish. The deterministic lint covers decision-space, condition-well-formedness, redline-template, approved-options, and referential integrity. An optional AI semantic lint (advisory, default-off) layers on top.

Error contract: failures raise PlaybookAuthoringError(payload, status=400) (playbook_authoring.py), surfacing as 400 {"error": "…"} with an optional "code" (e.g. playbook_draft_conflict). The lint is fail-open: if the lint itself throws, the publish is not blocked.

The contract. To change how NDAs are judged, you edit the Playbook — never the engine. The AI receives the playbook rules in its packet and is told to comply to them; it must never invent or override a rule.

The AI review pipeline

Review is on-demand only — there is no automatic AI review on inbound documents. A user clicks Review; the server enqueues an async job; the workstation polls for the result.

Entry point & on-demand contract

The review is triggered by POST /api/matters/<id>/review-refresh → handle_matter_review_refresh() in nda_automation/routes/matters.py. It enqueues a review and returns 202 immediately — it never runs the AI engine inline (asserted by tests/test_async_review_backend.py). Inbound Gmail matters import as "Not Reviewed" and are reviewed only when a human asks (server.py comment; tests/test_inbound_auto_review.py).

The packet

Before the model is called, the app assembles an assessment packet via build_ai_assessment_packet() in nda_automation/ai_assessment_prompt.py. It bundles: the clause requirements / playbook rules (playbook_rules_for_ai() in playbook_rules.py), the binding policy block (build_playbook_policy_block() in playbook_policy.py), the Stage-1 structure context (build_contract_structure() in contract_structure.py), and the source paragraphs (with omitted-paragraph counts + truncation markers). The model sees a flat {id, index, text} packet plus the rules.

The structure layer (Stage 1)

Before the packet is built, a deterministic Stage-1 pass turns the raw paragraph list into structure/reference/concept signals that ride into the packet as context. It is network-free — no model call — and feeds Stage 3 (clause review) a map instead of a flat blob.

  • Structure: build_contract_structure() (contract_structure.py) assembles the section/heading map from the paragraph list.
  • Headings: heading_detection.py (parse_leading_number et al.) detects numbered headings/leading numbers that seed the structure.
  • References: reference_resolver.py (resolve_document_references, build_reference_integrity_signal) resolves cross-references and emits an integrity signal.
  • Concepts: concept_classifier.py (classify_document_concepts) tags paragraphs/sections with concept labels — shared with the dashboard-search grouping.

The two models

Reviewer (first pass)
Default anthropic/claude-opus-4.8-fast — DEFAULT_OPENROUTER_MODEL at ai_review.py, selected by env NDA_AI_MODEL (AI_REVIEW_ENV_MODEL, ai_review.py). Resolved in configured_ai_assessment_reviewer() (ai_assessor.py).
Verifier (second pass)
Default deepseek/deepseek-v4-pro — DEFAULT_VERIFIER_MODEL at ai_verifier.py, env NDA_AI_VERIFIER_MODEL (ai_verifier.py). Enabled by NDA_AI_VERIFIER (opt-in). Resolved in resolve_verifier() (ai_verifier.py).

Both run through OpenRouter; the key (OPENROUTER_API_KEY) lives in backend config and is never exposed to the browser.

Local reviewer ≠ prod reviewer (deliberate divergence). .env.example ships NDA_AI_MODEL=x-ai/grok-4.3 for local dev (cheaper), while the code default (DEFAULT_OPENROUTER_MODEL, ai_review.py) and prod (render.yaml) both run anthropic/claude-opus-4.8-fast. So a local review can legitimately disagree with prod on borderline clauses — that's the model, not a bug. To mirror prod, set NDA_AI_MODEL=anthropic/claude-opus-4.8-fast in your .env.

The decision arbiter

Clause decisions are synthesised in three phases:

  1. AI assessment → build_ai_first_review_result() (ai_first_review.py) normalises reviewer assessments into clause results.
  2. Adversarial verifier → apply_ai_verifier() (ai_verifier.py). Escalated clauses (fail/review) are always verified; passes are verified only when confidence is below the high-confidence pass threshold (0.85, ai_verifier.py). A refute/uncertain verdict downgrades to review — the verifier never autonomously acquits a clause to pass.
  3. Reconciliation → _reconcile_unchanged_clause_review_states() (ai_first_review.py) and _reconcile_clause_decision() (ai_first_review.py) propagate changes into review_state and apply floor normalisations.

The semantic confidence floor is SEMANTIC_REVIEW_THRESHOLD = 0.75 in decision_arbiter.py — a pass below it is escalated to review. review_state.py (_normalize_clause_decision()) imports the same constant so the floor is single-sourced.

The verifier records the truth, not a phantom pass. When the verifier batch call fails or the response omits a clause, that clause is stamped unverified (reason verifier_unavailable on a total failure, verifier_missing_verdict on a per-clause omission) via _skip_record() (ai_verifier.py ~313–317) — it is never silently affirmed. ai_verifier_ran reflects reality: on a total failure (every record unverified) the summary reports status="failed" so downstream reads ai_verifier_ran = False (ai_verifier.py ~211–217); a partial failure stays completed but surfaces an unverified_count. It still fails open — the first-pass verdict stands and nothing is blocked purely because verification didn't run — but the record never claims a verification that did not happen.

Evidence grounding

Every pass/fail on a present clause must cite a real, verbatim quote. classify_grounding() (evidence_grounding.py) returns grounded / absence / ungrounded. Ungrounded findings are demoted by downgrade_ungrounded_finding() (evidence_grounding.py) → decision REVIEW, blocks_send=True, reason code ungrounded_finding. The two legitimate no-quote cases: a required clause FAILed as missing, and a prohibited clause that correctly does not appear. Wired in via build_grounding() (ai_first_review.py).

review_state

The per-clause review_state struct (review_state.py) carries state (pass|review|check), decision (pass|review|fail), blocks_send, blocks_auto_send, requires_human_review, and reason_codes. clause_review_state() sets it per clause; aggregate_review_state() (review_state.py) rolls up the document-level state.

Why it's trustworthy. Fail-closed everywhere: a missing assessment, an invalid citation, or a provider error escalates to Review and blocks auto-send rather than waving a document through. Low confidence becomes "ask a human"; uncertainty leans to Review.

Redlines & generation

Two write paths produce documents: redlining marks up a received NDA as Word tracked changes; generation builds a brand-new NDA deterministically from the template + playbook.

Redlines: propose → diff → anchor → export

  • Propose. A failed clause's suggested-fix wording comes from the playbook's redline_guidance; the FE edit model lives in static/js/modules/redline-edit-contract.mjs (action types REPLACE_PARAGRAPH / DELETE_PARAGRAPH / INSERT_AFTER_PARAGRAPH / FORMAT_PARAGRAPH).
  • Diff. Word tracked changes are built in nda_automation/redline_xml.py — the _source_tracked_replace_paragraph* / _source_tracked_delete_paragraph / _source_tracked_insert_paragraphs builders emit <w:ins> / <w:del> revision markup. redline_replace_uses_whole_text_markup() (redline_xml.py) decides whole-text vs char-level tokenised diff.
  • Anchor. source_index is non-unique provenance — split blocks (a paragraph cut on an internal blank line) share it on purpose. paragraph_index (a.k.a. index) is the unique blank-line-block key. Export and the health gate key on paragraph_index, falling back to source_index: see _expected_redline_source_index() (docx_health.py) and redline_source_index() (redline_edit_contract.py).
  • Export. Built in docx_export.py.

Coverage gates

The DOCX gate verify_export_content_coverage() (docx_health.py) asserts the export body is non-empty, covers ≥50% of source text (EXPORT_CONTENT_COVERAGE_RATIO = 0.5; the unrelated _COVERAGE_DIAGNOSTIC_WINDOW = 80 is only a log-truncation width, not the floor), and that the accepted-paragraph sequence matches the expected sequence built from source + redline edits. The PDF path can't do exact sequence comparison (reconstruction reflows), so verify_pdf_reconstruction_redline_coverage() (docx_health.py) instead finds each redline's target paragraph by confident text match and verifies w:ins/w:del markup is present, using one-to-one bipartite matching so identical redlines consume distinct paragraphs. Both fail closed.

Generation (deterministic)

POST /api/generate-nda (routed at server.py) delegates to the generation workflow. By default generation is deterministic: generation_ai_enabled() (nda_generation.py) reads NDA_GENERATION_AI_ENABLED; in production it is set to false, so no OpenRouter call is made and the document is built purely from the template + playbook. The template is nda_automation/templates/generic_nda.docx (Aspora = second party, counterparty = first), with Term/Survival and Confidential-Information exclusions realigned from the template defaults to the live playbook at generation time.

gen-verify (the ship gate)

Before any persistence, assert_generated_nda_is_on_position() (nda_generation.py) runs two independent screens: (1) self_check_generated_nda() (nda_generation.py) — a network-free deterministic oracle that re-extracts the generated text and runs review_nda(..., ai_enabled=False), requiring zero native-clause failures, plus a key-free stub check that non_circumvention does not fail; and (2) a prohibited-position scan over the whole document. Either failure raises NdaGenerationError and blocks the ship.

The governing-law lock

Governing law is fixed by the signing entity and cannot be overridden. In nda_generation.py, an override_id that differs from the entity default raises NdaGenerationError (input category); an override equal to the default is a harmless no-op. The manifest always records governing_law_overridden = False. This closes the divergence where an override could pull another entity's court into a signed NDA.

PDF / DOCX rendering & Approach C

"Approach C" converts a PDF-source NDA into a working DOCX at ingest time, so the whole redline/export pipeline operates on a real Word document instead of a reconstructed one on the fly. A vendored faithful-render path shows the original DOCX byte-for-byte in the browser.

Approach C: convert at ingest

  • convert_pdf_matter_to_docx() lives in nda_automation/pdf_ingest_conversion.py.
  • It is wired at ingest: ingestion_service.py → _convert_pdf_matter_at_ingest() → _persist_pdf_working_conversion() (ingestion_service.py) → convert_pdf_matter_to_docx(). Non-PDF matters are skipped (document_type guard). The conversion re-keys paragraphs to anchor by source_index and registers a role="working" artifact.
  • Fail-open: any error in the chain returns the original matter unchanged — conversion never hard-blocks ingest.

working_docx_ready & the /working-docx seam

  • working_docx_ready is set in the render payload at matter_render_job.py via matter_has_working_docx() (matter_render_job.py), which checks for a ROLE_WORKING artifact in the registry.
  • The route GET /api/matters/<id>/working-docx → handle_matter_working_docx() (routes/matters.py, wired at server.py) streams the working DOCX bytes via pdf_export_service.build_matter_working_docx_export().

The vendored faithful render (docx-preview)

For DOCX-source matters there is a byte-faithful browser render, vendored (no CDN):

  • static/vendor/docx-preview/docx-preview.min.js (+ LICENSE) — confirmed present in the deployed tree.
  • static/js/docx-faithful-render.js — lazily injects jszip + docx-preview on first render (retrying), exposes renderFaithfulDocx(container, options), and always returns { ok, reason } — never throws, degrading to reconstruction on any failure.
  • Flag: nda.faithfulDocxRender (localStorage key from static/js/config.js; gate faithfulDocxRenderEnabled() in docx-faithful-render.js). Default OFF — set localStorage["nda.faithfulDocxRender"] = "1" to enable.

reconstruct_pdf_to_docx (the low-level fallback)

reconstruct_pdf_to_docx() (pdf_docx_reconstruction.py) is the engine that turns PDF bytes into DOCX bytes (semaphore/slot backpressure, page-cap, owner-keyed LRU cache, subprocess timeout/RLIMIT hardening). It knows nothing about redlines. convert_pdf_matter_to_docx() is the high-level wrapper that calls it, re-keys paragraphs for anchor-by-index, and enforces an empty-body guard.

PDF extraction: pypdf vs PyMuPDF

Document structure/text extraction uses pypdf (extract_pdf_document() in pdf_text.py). PyMuPDF / fitz is used only for visual signals — a cheap image-count decompression-bomb guard (pdf_text.py, via get_image_info(), no pixel decode) and page rasterization for preview thumbnails. fitz is never used for structure.

Why the Dockerfile asserts fitz. A silent PyMuPDF ABI mismatch would break page-image rendering only at runtime. The build smoke-test (import fitz; fitz.open(), Dockerfile:43) fails the build loudly instead. See Deploy & Ops.

Scanned-PDF OCR fallback (default OFF)

A scanned/image-only PDF has no extractable text for pypdf to read. nda_automation/pdf_ocr.py is an opt-in vision fallback for exactly that case — it sends page images to a vision model to recover text. It is off by default and bounded so it can't run away on cost or latency:

Data model & security

The unit of work is a matter — one NDA and everything attached to it, persisted as a JSON record under NDA_DATA_DIR. Multi-tenant isolation is fail-closed.

The matter store

  • Module: nda_automation/matter_store.py. create_matter() (matter_store.py) mints matter_<uuid12> and writes $NDA_DATA_DIR/matters/<id>.json.
  • Locking: _locked_store() (matter_store.py) takes an in-process threading.RLock() (30s timeout) and a cross-process fcntl.flock() on matters.lock, so create/dedupe/update read-modify-write is serialised even across processes. The one historical bug was an unlocked dedupe pre-check; the real R-M-W path was always sound.
  • Dedup: _find_gmail_duplicate_unlocked() (matter_store.py) runs inside the lock — it indexes attachment id / case-folded filename / SHA256, gates each candidate through ownership, and confirms filename-only matches by hash. Atomic check+write closes the TOCTOU window.

Performance & observability (scale hardening)

The store is a single-process file store, so a growing corpus turned several hot paths O(store). Four changes keep them bounded:

  • Per-record list cache — list_matters() is cached under a fingerprint of every record file's (name, mtime, size), consulted/refreshed only while the store lock is held (so it observes a consistent set) and storing the UNFILTERED list — owner scoping is always applied per-call after, so the cache can never serve cross-tenant data. The blob is keyed per record filename, not one monolithic blob: a write-through patch updates the one written record's entry (O(one record)) instead of re-serialising the whole store (~330ms/write on a 47MB/1200-matter store → a few ms). NDA_DISABLE_MATTER_LIST_CACHE forces the uncached path for tests.
  • Timeline cap — matter_timeline was append-only and grew unbounded; capped_timeline() now clamps it to TIMELINE_MAX_EVENTS = 500 (newest kept, oldest dropped, with a marker event) on every write.
  • Corpus fingerprinting off the request path — dedup SimHash fingerprints are computed inline only under a FINGERPRINT_INLINE_COMPUTE_BUDGET per request; the remainder is computed by a single-flight background backfill (_start_fingerprint_backfill, corpus_index.py), so a large cold store never serialises O(store) SimHash computes onto one request.
  • Failure visibility — swallowed failures now log (module LOGGER.warning/error with the exception type) instead of failing silently — e.g. the PyMuPDF import path logs the exact module+message at ERROR on a genuine load failure rather than degrading rendering invisibly.

Artifacts

nda_automation/artifact_registry.py is a thin registry over a matter's documents. Roles (artifact_registry.py): original, redline, reviewed, generated, counter, sent, signed, and working (the Approach-C PDF→DOCX). Actors: counterparty, ai, human (or an entity id). The matter carries an artifacts list + a current_artifact_id pointer.

Fail-closed ownership

_matter_owner_matches() (matter_store.py) is the tenant guard, used by every matter op (list/get/update/delete/export/dedup):

  • Empty caller owner_user_id = single-tenant/no-auth → wildcard (all matters in scope).
  • Non-empty caller = authenticated → matter's owner_user_id must match exactly.
  • A matter with no owner field (legacy / pre-ownership Gmail import) is not a wildcard — it fails closed and is never served to an authenticated caller with a different owner.

NDA_ADMIN_USERSAdmin authority list (http_auth.py). Entries are google:<sub> — the OAuth sub, not the email.
NDA_ALLOWED_HOSTSComma list of allowed request hostnames (your Render host); the host-check guard.
Allowlist fail-safe (know this before you deploy). With both NDA_ALLOWED_EMAIL_DOMAINS and NDA_ALLOWED_EMAILS unset/empty, the allowlist is OFF — every OAuth-verified Google identity is allowed (no lockout on a fresh deploy). Set either one and it flips to fail-closed: an email matching neither list is denied, and an existing session whose email no longer passes is treated as unauthenticated. Basic-auth sessions are never routed through the allowlist.
Admin is the OAuth sub, not the email (footgun). NDA_ADMIN_USERS entries take the form google:<sub> (the stable Google subject id), e.g. google:101508195488490085718 — not google:alice@aspora.com. For Google callers request_is_admin() also accepts the OAuth-verified email, but the canonical, stable id is the sub. Basic-auth usernames never inherit admin from an email.

Auth, admin & CSRF

  • Auth: _authorize_request() (server.py) — Google OAuth session first, then HTTP Basic (NDA_AUTH_USERNAME/NDA_AUTH_PASSWORD), else 401. Loopback hosts (127.0.0.1/::1/localhost, _is_loopback_host in http_auth.py) are auth-exempt for local dev.
  • Admin: request_is_admin() (http_auth.py). Authority = the NDA_ADMIN_USERS env list (matched against the verbatim user id, e.g. google:<sub>, and — for Google callers only — the OAuth-verified email), plus an in-app persisted admin list (Google callers only). Basic-auth usernames never inherit admin from an email.
  • CSRF: origin_allowed_for_request() (csrf.py) runs for state-changing methods only, matching the Origin (or Referer) host against an allow-set. Enforcement flag NDA_ENFORCE_CSRF (default ON for non-loopback binds, OFF for localhost). Checked before auth in do_POST/do_DELETE.
  • Rate limit: rate_limit.py + _rate_limit_request() (server.py) — bucketed by user id then real client IP (proxy-aware via NDA_TRUSTED_PROXY_COUNT), 300/min general, 240/min render, 60s window, 429 + Retry-After.

Gmail intake · DocuSign · Outbound

Three external surfaces: the Gmail poller that auto-imports inbound NDAs, the DocuSign e-signature flow, and the outbound send paths (with a hard recipient-confirmation gate).

Gmail intake

  • Scheduler: _gmail_sync_scheduler_loop() (server.py), a daemon thread; each tick runs _run_scheduled_gmail_sync() (server.py). Master kill-switch env NDA_GMAIL_SYNC_ENABLED (server.py).
  • Fetch query (fixed structure) + sender exclusion: _inbound_envelope_query(90) (gmail_integration.py) → in:inbox has:attachment (filename:docx OR filename:pdf) -from:me newer_than:90d, with a -from: exclusion clause for noise senders appended to the query. The default excluded set (DEFAULT_GMAIL_INBOUND_EXCLUDED_SENDERS, app_settings.py) is the e-sign platforms (docusign.net/.com, adobesign.com, documents.adobe.com, echosign.com, hellosign.com, pandadoc.com) plus calendar-notification@google.com; it's admin-editable, but DOCUSIGN_NOTIFICATION_DOMAINS is a hard floor that always applies. Window default GMAIL_INBOUND_WINDOW_DAYS = 90 (admin-configurable). Per-poll limit default 20, hard cap 40 (_MAX_GMAIL_IMPORT_LIMIT_CLAMP, gmail_integration.py; env NDA_GMAIL_IMPORT_LIMIT).
  • Deterministic pre-gate (before the paid model): a content scorer drops obvious non-NDAs (proposals, invoices, decks, questionnaires) before a paid intake AI call, cutting intake-AI spend markedly. A mandatory escape hatch (_metadata_has_explicit_nda_signal / the filename+text NDA signal, gmail_integration.py) forces any document with an explicit NDA signal through to the AI regardless — including foreign-language NDA titles. Detection uses a fixed built-in vocabulary; an opt-in custom-terms extension is gated behind NDA_GMAIL_CUSTOM_TERMS_ENABLED (default off; _custom_detection_terms only adds content-scorer signals — never widens the fetch query).
  • Executed-NDA capture: e-sign platform notification mail is usually terminal-dropped, but a completion email is often the only copy of an executed NDA. When such a platform email carries an explicit NDA signal (_esign_notification_nda_hit, gmail_integration.py), it's let through the pipeline but clamped to at most the triage lane and stamped with the esign_notification_nda provenance (ESIGN_NDA_CAPTURE_TRIAGE_REASON) — never a silent auto-import. Gated by NDA_GMAIL_ESIGN_NDA_CAPTURE (default on; off restores the unconditional drop).
  • Intake classifier (NDA vs not): classify_intake_attachment() (gmail_intake_classifier.py); model env NDA_GMAIL_INTAKE_MODEL, default deepseek/deepseek-v4-flash (gmail_intake_classifier.py). Additive — any error/timeout/cap falls back to the deterministic lane classifier.
  • Attachment selector / triage: gmail_attachment_selector.py; model env NDA_GMAIL_TRIAGE_MODEL, default deepseek/deepseek-v4-pro. Injection defense: the model's selection is intersected with the real candidate attachment ids (gmail_attachment_selector.py) — an injected instruction can't import an attachment that was never a candidate.
  • Review is deferred. import_inbound_matters() (gmail_integration.py) fetches, triages, and classifies but does not enqueue AI review — inbound matters land "Not Reviewed" and are reviewed only on demand.

Google Drive filing

Drive auto-files matters into a connected Google Drive folder. Handlers live in nda_automation/routes/drive.py; the client seam is nda_automation/drive_integration.py; settings persist through nda_automation/app_settings.py.

The master on/off gate is drive_paused — NOT the legacy enabled field. Every Drive call-site gates on drive_connected(owner) AND drive_active(), where drive_active() is simply not drive_paused() (app_settings.drive_active() / drive_paused()). The gate never reads enabled.
  • Why it's a distinct flag. Legacy connected users can carry a stale enabled:false on disk (written by an old normalizer). If the gate reused enabled, those working users would be silently switched off. drive_paused defaults false and is set only by the new admin pause toggle — so a never-paused connected user always keeps Drive.
  • The gate call-sites: matter_lifecycle.py (~640: connected = drive_connected(owner), active = drive_active()), drive_integration.py (~754–757), and every mutating route in routes/drive.py (~234/336/415: if not drive_connected(...) then if not drive_active()).

DocuSign

nda_automation/docusign_integration.py holds the client. HttpDocuSignClient creates envelopes (create_envelope()), reads status (get_envelope_status()) and recipients (get_envelope_recipients()). Signature placement uses anchor tabs built in _tabs_for(signer) — signHereTabs + dateSignedTabs anchored to document strings. Two signers: the counterparty and the single Aspora signer (NDA_DOCUSIGN_ASPORA_SIGNER_NAME/_EMAIL). OAuth env (docusign_connection.py): NDA_DOCUSIGN_CLIENT_ID, _CLIENT_SECRET, _OAUTH_REDIRECT_URI, _AUTH_SERVER (demo|prod), _CONNECT_HMAC_KEY.

Envelope document = current reviewed DOCX (P0 invariant). docusign_workflow.py never signs the un-redlined original when a matter has reviewer edits. The base document selection walks _DOCUMENT_PRECEDENCE — order reviewed → generated → sent → original (ROLE_REVIEWED is the first element, not excluded). The two paths differ:
  • Edited matter: the reviewed DOCX is rebuilt from the matter's CURRENT state via the same build_reviewed_docx path (and the same coverage gates) the download + approval handlers use — not a stored role=reviewed artifact, which could be stale after a reversal. If that rebuild can't be produced, the send is refused ("Refusing to send the un-redlined original for signature.") rather than falling back to the original.
  • Clean / no-edits matter (comment-only or nothing to redline): the selection is made with an explicit exclude_roles=(ROLE_REVIEWED,) argument at the call site, so precedence skips the reviewed role and falls to generated/sent/original. The exclusion is a call-site argument, not a property of the precedence constant.
Either way, the user-facing invariant holds: it never signs a stale reviewed copy or the un-redlined original of an edited matter.

Outbound (send redline / send document)

  • Send redline: handle_gmail_send_redline() (routes/gmail.py, wired at server.py).
  • Send document: handle_send_document() (routes/send_document.py, wired at server.py) — reuses the same Gmail send plumbing.
  • Recipient-confirmation gate: require_confirmed_recipient() (gmail_matter_outbox.py), reached via validate_outbound_send_ready() before any Gmail call. It raises RecipientConfirmationError → 400 if the recipient came from an inbound header and was not confirmed, if the confirmed value is empty, or if it doesn't match the resolved recipient ("The confirmed recipient does not match the NDA recipient; refusing to send."). This blocks spoofed-Reply-To exfiltration.

Gotchas / known hazards

The things that bite a fresh contributor on day one. None are bugs — they are load-bearing constraints of the single-process, no-build-step design.

1 · Single-process server: CPU-heavy background work starves requests

The server is one ThreadingHTTPServer process with no WEB_CONCURRENCY. A CPU-heavy background job — a big Gmail extraction sweep or a corpus fingerprint backfill — holds the GIL and starves request threads, so even static-asset serving stalls and the app can look wedged. This is why the intake loop yields cooperatively (NDA_GMAIL_SYNC_YIELD_MS) and fingerprinting is capped inline + pushed to a single-flight backfill (see Data Model). If the UI goes unresponsive, suspect a background CPU spike before a crash.

2 · The ?v= manifest refuses to record changed bytes under an unchanged token

python -m nda_automation.static_versioning --write will REFUSE to record an asset whose bytes changed while its ?v= token stayed the same — the whole point of the guard is that a bytes-change must ride a token bump. So when you change any .js/.css asset you MUST: bump its ?v= in index.html first, then --write. .html files are excluded from the manifest (they always revalidate), so they need no token there.

3 · Never run the whole backend suite in one process (and never in parallel)

A monolithic pytest tests/ SIGKILLs partway through under memory pressure on this box, and concurrent full-suite runs across worktrees gridlock the shared machine. Run targeted files or small fresh-process batches with python -m pytest -q -p no:cacheprovider <files> — full detail in the Run the tests recipe.

4 · Editing an in-app guide requires bumping its own ?v= in app.js

The Guide tab loads each guide through an <iframe> whose src comes from GUIDE_SRC in static/app.js (e.g. developer: "/static/developer-guide.html?v=…"). Editing developer-guide.html alone is invisible to a loaded browser — the iframe keeps serving the cached copy until you bump the guide's ?v= token in GUIDE_SRC (then run the manifest ritual, since app.js itself changed). This is separate from the asset-token ritual in Architecture.

How-to recipes

Common changes, with the exact place to make them.

Add a clause type

  1. Add the clause object to the "clauses" array in playbook.json (id, name, type required|prohibited, requirement, rules block, search_terms, taxonomy_groups). The checker indexes it on load.
  2. Native vs dynamic: native clauses have a deterministic engine checker; dynamic clauses (e.g. non_circumvention) are only emitted by the AI engine — the deterministic engine never produces them, so a dynamic clause needs both the playbook entry and an AI-engine seam (and any test must add a backend seam, not just a FE change).
  3. Validate via POST /api/playbook/draft, then publish via POST /api/playbook/publish (lint-gated, error-severity blocks).

Change an AI model

Set the env var and redeploy (or flip live in Render):

JobEnv varDefault

Change a playbook rule

Edit the rule in the Playbook, never the engine. The AI receives the playbook rules in its packet; deterministic checkers read the same playbook. Hardcoding a rule the playbook owns is a "deterministic ghost" — don't.

Debug a stuck review

  1. Remember review is async — POST /api/matters/<id>/review-refresh returns 202; the result arrives later. "Reviewing…" is normal up to ~4 min; a stale recorded start (~10 min) reads as "stalled", not failed.
  2. Only a recorded error shows "failed". Check the provider key (OPENROUTER_API_KEY), NDA_AI_REVIEW_ENABLED, and the review-state reason codes.
  3. Grounding demotions (ungrounded_finding) and the 0.75 confidence floor push borderline clauses to Review by design — that's not a bug.

Run the tests

Do not run the whole backend suite in one process on this box. A monolithic pytest tests/ (all ~210 test files in one interpreter) climbs memory until the OS SIGKILLs it partway through — a partial run that reads as a spurious failure. And never run the full suite in parallel across worktrees: concurrent full-suite runs gridlock the shared box.

The reliable workflow is targeted files, or the full suite in small fresh-process batches — never all files in one process:

Targeted (the default): python -m pytest -q -p no:cacheprovider tests/test_ai_verifier.py tests/test_async_review_backend.py
A batch of the full suite: run a handful of files per invocation, each a fresh process, e.g. python -m pytest -q -p no:cacheprovider <first ~15 files> then repeat for the next batch.
Frontend: npm run test:frontend:utils && npm run test:frontend
Lint: ruff check nda_automation tests

-p no:cacheprovider keeps a batch run from writing a stale cross-batch cache. There is no Makefile; pytest + npm run directly. The pytest suite includes the review eval gate (counsel-style review cases). See Gotchas / known hazards for why the single-process ceiling exists.

The constraint is this box, not the suite. CI (.github/workflows/ci.yml) runs the full python -m pytest -q on a clean ubuntu runner across Python 3.9 / 3.11 / 3.12 — the whole suite in one process is fine on a fresh machine; it's only this shared dev box that SIGKILLs a monolithic run under memory pressure. CI has three jobs: test (lint + compile + the OAuth-focused slice + full pytest, matrixed over the three interpreters), frontend, and docker-build. The docker-build job is the one that catches prod-only dependency breaks — a camelot/PyMuPDF ABI or co-resolution failure that only shows up in the full .[pdf,gmail,tables] Docker image — so a green local test run is not a substitute for it.
