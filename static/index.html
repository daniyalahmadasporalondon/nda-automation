    Structure, references, and concepts

    1

    Ingest

    Manual upload, repository matter, or Gmail intake provides the source document and metadata.

    2

    Extract

    DOCX/PDF extraction produces plain text plus source paragraph records with Word numbering, heading style, table, and supplemental-part metadata.

    3

    Align

    Paragraph IDs, indexes, offsets, and provenance are assigned before any clause decision runs.

    4

    Map

    The contract structure map detects document-specific headings, numbering, ranges, aliases, and a resolver-ready reference index.

    5

    Resolve

    Explicit clause, article, section, schedule, annex, and appendix references are linked to detected document sections.

    6

    Classify

    Deterministic concept tags identify confidentiality, use, disclosure, return, survival, and carve-out concepts by paragraph and section.

    7

    Check

    Python clause checks apply playbook rules to the actual reviewed paragraphs and evidence.

    8

    Verify

    Evidence provenance validation ensures matched text still resolves to the reviewed source.

    9

    Redline

    Redline edits, comments, export decisions, and selected templates are prepared for DOCX output.

    checker visibility

    Clause decision logic

    checker visibility

    How clause decisions become pass, review, or check

    Auditable outputs

    Decision reading order

    1. 1review_state

      Start with pass, review, or check plus send/redline blocking.

    2. 2reason_code

      Read the machine-readable cause before interpreting prose.

    3. 3structured_evidence

      Inspect paragraph IDs, matched terms, signal type, rule bucket, and counted flag.

    4. 4analysis object

      Open the checker-specific object, such as term_survival_analysis or governing_law_analysis.

    5. 5audit_trace

      Follow input, evidence, signal classification, analysis outputs, and final decision.

    Reason-code taxonomy

    Pass
    approved_governing_law, term_survival_within_cap, no_non_circumvention_restriction
    Review
    weak_mutuality_signal, usage_right_language_needs_review, unresolved_survival_reference
    Check
    missing_required_clause, unapproved_governing_law, prohibited_non_circumvention_restriction
    Semantic boundary
    semantic_confidence_below_threshold marks a deterministic/semantic confidence issue for human review.

    Hardening guards

    False clear guard
    Term and Survival does not pass on unrelated language such as "Claims survive for three years."
    False check guard
    Non-Circumvention ignores negated references like "may not include non-solicitation obligations."
    Attachment guard
    Confidential Information checks whether independent-development qualifications actually attach to the carve-out.
    Payload fields
    Admin should reconcile reason_code, structured_evidence, analysis object, and audit_trace before changing a rule.

    Mutuality

    Module
    nda_automation/checks/mutuality.py
    Purpose
    Confirm the NDA creates reciprocal confidentiality obligations for both parties, not a one-way receiving-party model.
    Inputs
    Reviewed paragraphs, role definitions, reciprocal-obligation terms, one-way terms, and shared structure context.
    Pass
    Strong reciprocal obligation language binds both parties as disclosing and receiving parties.
    Review state
    Weak mutuality labels or role definitions without operative reciprocal obligations.
    Check
    One-way or unilateral language fixes only one side as receiving party or otherwise fails mutuality.
    Audit object
    mutuality_analysis
    State object
    review_state normalizes this checker into pass, review, or check and records send/redline blocking behavior.
    Reason codes
    reason_code and reason_codes identify the machine-readable cause for pass, review, or check.
    Structured evidence
    structured_evidence records expose paragraph provenance, matched terms, signal type, rule bucket, counted flag, and reason.
    Audit trace
    audit_trace records the normalized checker steps, evidence summary, analysis outputs, and final decision reason.
    Redline behavior
    Check decisions use the mutuality redline template; review decisions do not auto-redline.
    Human-review boundary
    Definitions alone are not enough for pass; they are treated as human-review evidence.

    reference resolver

    Cross-reference resolution

    reference resolver

    How explicit cross-references are resolved

    Backend first

    Inputs

    Reviewed paragraph records plus the Contract Structure Map's section IDs, paragraph ownership, aliases, and resolver-ready reference index.

    Supported references

    Clause, article, section, schedule, annex, annexure, appendix, and plural references such as "Clauses 1, 1A and 2".

    Resolution method

    Each detected reference is matched against the document-specific alias index, then marked resolved, partial, or unresolved.

    Outputs

    Reference text, source paragraph, source section, target section IDs, unresolved numbers, target labels, and summary counts.

    No fixed numbering assumption
    The resolver uses the detected structure of the current document, including DOCX list numbering when present, so Clause 4 or Article II means whatever this NDA actually mapped it to.
    False-positive guard
    Heading self-references are skipped and prose fragments like "this section introduces" are not treated as Section I references.
    Current limitation
    Explicit references are resolved and deterministic concepts are classified. Semantic phrases like "the confidentiality obligations above" still need a future AI evaluator.
    Checker handoff
    Term and Survival now uses resolved references and concepts to inspect what actually survives instead of trusting the survival sentence alone.

    concept classifier

    Deterministic concept tagging

    concept classifier

    How deterministic concepts are tagged

    Rules first

    Inputs

    Reviewed paragraphs plus detected section headings and paragraph ownership from the Contract Structure Map.

    Concepts

    Confidentiality definition, confidentiality obligation, use restriction, permitted disclosure, return/destruction, survival, and carve-out concepts.

    Section rollup

    Paragraph concepts are rolled up to their owning section, so referenced clauses can be judged by substance.

    Outputs

    Concepts by paragraph ID, concepts by section ID, matched signals, and summary counts for audit.

    Term checker use
    If survival says "Articles 2, 3, 4 and 5 survive," the checker can inspect whether those articles are confidentiality provisions.
    Auditor visibility
    Classifier output is included in the review payload as concept_classifier and can be inspected alongside references.
    Boundary
    This is deterministic pattern classification, not AI interpretation. Ambiguous semantic references remain a later evaluator step.

    ai semantic review

    AI review methodology

    ai semantic review

    How AI-first review works

    Optional layer

    Activation

    Admin can save a local OpenRouter API key and turn AI on from the Admin tab, or deployment can use NDA_AI_REVIEW_ENABLED, NDA_AI_PROVIDER, NDA_AI_MODEL, and OPENROUTER_API_KEY in the backend environment.

    Scope

    AI-first is the active review engine for Mutuality, Confidential Information, Governing Law, Term and Survival, and Non-Circumvention. Deterministic review remains internal-only for explicit generation checks. Signatures remain separate.

    Inputs

    AI-first assessment uses clause requirements, playbook rules, structure context, and selected source paragraphs.

    Outputs

    ai_review records the run summary. ai_review_analysis records each provider decision, confidence, cited spans, validation status, and disagreement state.

    Provider
    OpenRouter through nda_automation/ai_review.py using structured JSON output. API keys are read from backend configuration or ignored local app data and are never exposed to browser JavaScript.
    Default model
    x-ai/grok-4.3 by default; overridable with NDA_AI_MODEL in the backend environment.
    Confidence gate
    Default threshold is 0.75. Lower-confidence AI output becomes review rather than pass.
    Citation validation
    Pass and fail decisions must cite paragraph IDs that exist and quote text that appears in the reviewed source paragraph.
    Merge policy
    Missing assessments, invalid citations, provider errors, or configuration errors fail closed or escalate to review and block auto-send.
    Audit fields
    Review payloads expose ai_first_review, ai_review, ai_first_assessment, structured_evidence, proposed_redline, and evidence_trust.
